# 5分钟实操讲稿（网页+Agent）

## 开场

下面我用 5 分钟，带大家走一遍最基础的一套操作。今天我们不装 GitHub Desktop，也不讲命令行，只讲 GitHub 网页和 Agent 怎么配合。

## 第一步：先看状态

先打开 GitHub 仓库页面。这个时候不要急着改文件，先看两件事：第一，我现在进的是不是对的仓库；第二，我当前看的主线是不是 `main`。如果对项目还不熟，最简单的做法不是自己猜，而是先对 Agent 说一句：“帮我看看这个仓库当前是什么状态。”这样你至少会知道现在是不是有人刚改过、主线是不是稳定、自己接下来适不适合直接开始动。

## 第二步：先开分支，不直接动主线

如果你这次改动只是一个新尝试，比如首页视觉、文案方向或者某个组件表现，就不要直接在 `main` 上改。直接在 GitHub 网页顶部找到分支下拉菜单，输入一个新的分支名字，然后创建分支。你也可以先问 Agent：“我这次任务适合直接改，还是应该先开分支？”对设计师来说，这一步非常重要，因为分支本质上就是你的安全试验区。

## 第三步：改文件

分支创建好以后，找到你要改的文件，点击右上角的编辑按钮，进入 GitHub 网页编辑器。这个时候你就可以直接修改文案、说明文档，或者一些轻量文件内容。改完以后，不要急着提交，先看一下这次改动是不是清楚、是不是只围绕一个主题。如果你不确定，也可以先问 Agent：“帮我看看我这次改动是不是只讲了一件事。”

## 第四步：写清楚提交说明

确认改动没问题以后，就要写提交说明。这里最忌讳写成 `update`、`fix`、`改一下` 这种模糊的话。你可以直接让 Agent 帮你整理，比如说：“帮我根据这次改动写一个清楚的 commit message。”好的提交说明，应该让团队一眼看懂你到底改了什么。

## 第五步：发起 PR

如果这次改动已经完成，就不要把“我改完了”和“可以直接进主线了”画等号，而是先发起 Pull Request。PR 的作用就是把你的改动变成一份正式提案，让团队 review 之后再决定要不要合并。这里你也可以继续让 Agent 帮忙，比如说：“帮我写一段 PR 描述，说明我改了什么、为什么改、希望大家重点看哪里。”

## 最后一句

所以整套最小操作其实很简单：先看状态，再开分支，再修改，再提交，再发 PR，最后再合并。你真正需要学会的，不是先记命令，而是先把目标讲清楚。
