# 面向设计师的 Git 与 Agent 实操教学稿（无 Desktop）

## 这版适合谁

这版适合：

- 不想安装 `GitHub Desktop`
- 不想从命令行学起
- 希望像在 Codex 里一样，用自然语言完成大部分操作
- 需要知道 Git 是什么，但不想被术语淹没

一句定位：

> Git 是底层的版本机制，Agent 是你和 Git 打交道的自然语言入口。

---

## 最适合设计师的开场说法

可以直接这样讲：

> 对设计师来说，Git 不必被理解成“写代码工具”，它更应该被理解成“项目版本和协作系统”。  
> Agent 则像一个会说人话的协作助手。  
> 你不需要先记命令，再去操作 Git；你只需要先说清楚目标，Agent 再帮你完成具体步骤。

再补一句：

> 所以今天这节课，我们不把重点放在背命令上，而是放在理解流程、会描述目标、知道每一步在保护什么。

---

## 这一套的核心观点

### 对设计师来说，真正要学的不是命令，而是 4 个问题

1. 我现在在改什么
2. 我这次改动要不要和主线隔开
3. 我怎么把这次改动说清楚
4. 我什么时候可以把它合回主线

### 对 Agent 来说，默认交互应该是自然语言

不是：

- `/new-branch`
- `/commit`
- `/push`

而是：

- “帮我开一个新分支，我想试试首页改版。”
- “帮我看看我刚才改了哪些文件。”
- “把这次修改整理成一个清楚的提交说明。”
- “检查一下这次改动适不适合合回 main。”

---

## 设计师需要懂的最小 Git 概念

### `repository`

项目仓库。  
就是整个项目的版本空间。

### `branch`

分支。  
就是把你的尝试先放到一条支线里，不直接碰主线。

### `commit`

一次正式存档。  
是“带说明的保存”。

### `pull request`

正式提案。  
表示“我做完了一个方案，请团队 review 后决定是否合并”。

### `main`

主线版本。  
可以理解成当前默认的正式版本。

---

## 设计师版一句话翻译

- `branch` = 方案支线
- `commit` = 带说明的存档
- `pull request` = 提案单
- `merge` = 正式采纳

一句总结：

> branch 是方案，commit 是存档，PR 是提案，merge 是采纳。

---

## 这堂课怎么讲最顺

推荐顺序：

1. 先讲 Git 是管理变化，不是管理代码
2. 再讲 Agent 让人不用先学命令
3. 然后做一次自然语言驱动的完整练习
4. 最后补 GitHub 网页上的基本动作

不要一开始就讲：

- 终端
- `rebase`
- `cherry-pick`
- 复杂冲突处理

---

## 实操课总目标

这堂课只要求学员完成下面这条闭环：

1. 看懂当前项目状态
2. 说出“我要开分支做一个尝试”
3. 完成一次小改动
4. 让 Agent 帮忙整理提交说明
5. 发起一次 PR
6. 理解为什么这套流程更安全

---

## Part 1：先把话说顺，5 分钟

可以直接照着讲：

> 以前大家理解 Git，常常是“要学很多命令”。  
> 但在 Agent 时代，设计师更合理的使用方式不是去背命令，而是先学会描述任务。  
> 你要知道自己是在“改主线”“试方案”“做提交”“发提案”，然后把这个目标说给 Agent 听。

再说一句：

> 所以今天我们学的不是命令，而是协作语言。

---

## Part 2：第一轮实操，用自然语言理解项目状态，10 分钟

### 练习目标

让学员先学会问，而不是先学会点。

### 可以现场演示的提问

#### 提问 1：先看项目状态

> “帮我看看当前项目现在是什么状态。”

你可以解释它背后的真实目的：

- 我有没有未提交的改动
- 我现在在哪个分支
- 最近有哪些变化

#### 提问 2：看最近变化

> “帮我总结一下这个项目最近改了什么，用设计师能听懂的话说。”

这个问题特别适合新加入项目的人。

#### 提问 3：看风险

> “如果我现在直接改主线，会有什么风险？”

这一步是为了建立分支意识。

---

## Part 3：第二轮实操，用自然语言开分支，10 分钟

### 你要教给学员的不是按钮，而是判断

判断标准很简单：

- 小修文案，可以直接在当前安全分支上改
- 新视觉尝试，应该开新分支
- 任何可能影响主版本的探索，都先不要直接碰 `main`

### 可以直接演示的说法

#### 场景 1：准备试首页改版

> “帮我开一个新分支，用来做首页视觉改版尝试。”

#### 场景 2：准备改 onboarding 插图

> “我想单独试一下 onboarding 的插图方向，帮我新建一个分支，不要影响主线。”

### 你要顺手讲清楚的点

> 分支不是高级玩法，它是保护自己和保护主线的基本手段。

---

## Part 4：第三轮实操，用自然语言完成一次小改动，10 分钟

这里不强调学员自己敲 Git 命令。  
重点是让他们知道，做完改动后下一步该怎么描述。

### 可以演示的说法

#### 场景 1：改了一个 README 或说明文档

> “我刚改了项目说明，帮我总结这次具体改了什么。”

#### 场景 2：改了一个页面视觉

> “我刚更新了首页头图和标题层级，帮我整理成一次清楚的小提交，不要和别的改动混在一起。”

#### 场景 3：不确定改动是不是太杂

> “帮我判断一下，我这次改动是不是应该拆成两次提交。”

这一步最重要的教学点是：

> 设计师不用先会写标准命令，但要学会把改动意图说清楚。

---

## Part 5：第四轮实操，用自然语言整理 commit，5 分钟

### 不要教学员写这种提交说明

- `update`
- `改一下`
- `fix`

### 直接教他们这样问

> “根据我这次改动，帮我写一个清楚的 commit message。”

### 再给两个更具体的例子

> “这次主要是更新首页文案和按钮层级，帮我写一个适合团队看的提交说明。”

> “我这次只动了 onboarding 的图标资源，帮我写一个简短但明确的 commit message。”

### 你可以顺口解释

> commit message 不是写给 Git 看的，是写给团队和未来的自己看的。

---

## Part 6：第五轮实操，发起 PR，10 分钟

这一步建议保留 GitHub 网页，因为它对设计师最直观。

GitHub 官方支持直接在网页上新建仓库、编辑文件和创建 Pull Request。  
也就是说，就算不装 GitHub Desktop，也可以完成基本协作。

参考：

- [Creating a new repository](https://docs.github.com/articles/creating-a-new-repository)
- [Editing files](https://docs.github.com/articles/editing-files-in-another-user-s-repository)
- [Creating a pull request](https://docs.github.com/articles/creating-a-pull-request?tool=webui)

### 可以这样讲 PR

> Pull Request 不是“程序员审批单”，它本质上是一张协作提案。  
> 你在上面告诉团队：我做了什么、为什么这么做、请大家看一下，没问题就合并。

### 可以演示的自然语言

> “帮我把这次修改整理成一段 PR 描述，重点说明我改了什么、为什么改、希望大家重点看哪里。”

### 适合设计师的 PR 描述结构

1. 这次改了什么
2. 为什么改
3. 希望大家重点看哪里
4. 有没有风险或待确认项

---

## 如果完全不用 GitHub Desktop，最低操作组合是什么

最小组合可以是：

1. `GitHub 网页`
2. `Agent / Codex`

分工可以这样理解：

- GitHub 网页：看文件、看 PR、发 PR、简单改文件
- Agent：解释项目状态、帮你判断是否该开分支、整理 commit 和 PR 文案、辅助检查风险

一句总结：

> 网页负责可视化，Agent 负责理解和辅助决策。

---

## 如果要像 Codex 一样使用，应该怎么教

你可以直接定一条原则：

> 默认先说目标，不先想命令。

### 推荐学员常用的 8 句自然语言

1. “帮我看看这个项目现在的状态。”
2. “帮我总结最近的改动。”
3. “这次需求应该直接改，还是先开分支？”
4. “帮我开一个分支，名字要清楚一点。”
5. “帮我检查我这次改了哪些文件。”
6. “帮我把这次改动整理成一个提交说明。”
7. “帮我写一段 PR 描述。”
8. “帮我判断这次改动能不能安全合回 main。”

---

## 对设计师最重要的判断句

如果你只想让学员记住一句，那就记这个：

> 我不需要先会 Git 命令，我需要先会描述协作目标。

因为真实工作里更常见的问题不是“不会敲命令”，而是：

- 不知道该不该开分支
- 不知道改动该不该拆开
- 不知道怎么解释给团队
- 不知道什么时候能合并

这些都更像协作问题，不是语法问题。

---

## 一段可直接照读的讲稿

> 今天我们不把 Git 当成程序员工具，而是把它当成团队协作系统。  
> 对设计师来说，最重要的不是学很多命令，而是知道什么时候该开分支、怎么描述一次改动、怎么把一个方案作为提案发出去。  
> 在 Agent 的使用场景里，这件事会更简单，因为你不需要先背命令，你可以直接说：帮我看状态、帮我开分支、帮我整理提交、帮我写 PR。  
> 也就是说，Git 仍然存在，但它不再要求你先适应机器语言；相反，是 Agent 来适应人的语言。

## 一页结尾总结

### Git 对设计师的价值

- 记录变化
- 保护主线
- 支持试错
- 让协作更清楚

### Agent 对设计师的价值

- 降低门槛
- 不用背命令
- 帮你解释状态
- 帮你整理提交和 PR

### 最重要的转变

不是“学命令”。  
而是“学会用自然语言表达协作目标”。
